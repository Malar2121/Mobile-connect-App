Abstract
Today's families are more dispersed than ever, and the apps that are used by the family members to communicate among themselves lack integration; family interactions happen on generic messenger apps, arrangements on generic calendar apps, and photographs are distributed across their personal devices and hosting services. There is not yet a platform that would enable all that communication, arrangement-making, storage, and security in the private space designated for the particular family. And this is what the purpose of the project was - to develop this kind of platform as Family Connect – a cross-platform mobile application for the private family-invite-only space.
Family Connect is a fully functional piece of software product. React Native app created using Expo SDK 54 is connected with the backend (Node.js + Express) using REST API with ten route groups, and Socket.io provides the live chat, typing, read receipts, location share, and SOS notifications. MongoDB Atlas hosts database, media stored on Cloudinary, and push notifications are provided by Firebase Cloud Messaging. Family ID filters all the queries about family data in the server, and user authentication uses password hashing and JWT refresh token technology. Client-side has numerous accessibilities like three types of interface modes (normal, for minors, and elders), high contrast themes, low motion mode, and three languages (English, Tamil, Sinhala).
The entire system has been tested with 180 documented test cases, 122 REST and 7 Socket.io, the latter being run against the running server – 170 of them succeeded (94.4%). Ten user acceptance scenarios have been accepted. The backend has been deployed on Railway, and the mobile client has been built with Expo Application Services. The project demonstrates how to create a production-level system for coordinating the activities of a family within an academic year and indicates further steps for improving the software.

1  Project Charter
In this chapter, the problem statement that the project seeks to solve is outlined, and stakeholders in the project and their services are identified along with the boundaries within which the system has been created. It also describes the assumptions, limitations, goals, and requirements that have been defined at the proposal stage to guide the project throughout the year.
1.1  Problem Statement
Managing family life from household to household can be difficult. Relatives are apart from each other both by virtue of distance and their own schedules, and there isn’t always an established meeting place to organize plans. Birthday announcements are made through one chat group while the location is discussed on another and the family member who doesn’t see either message doesn’t show up. Photos and videos taken at the actual events get scattered through WhatsApp chats, Facebook posts, and phone galleries, thus preventing the family’s shared history from being organized in one private place.
This is not solved through general-use services as they weren’t designed specifically for use by families. While messaging platforms help manage the conversation itself, they lack any formal way of handling events, RSVPs, and scheduling polls. Calendaring apps help schedule the events but lack personality and are outside the family’s communication process. Photography services are perfect for storing media but have no concept of the events that media was taken at. Besides the fragmentation problem, the families that use social media give up privacy that some members are willing to give.
Family Connect takes on the challenge and introduces an application on the mobile phone that includes event planning with availability polls, live family chat, private family album of photos and videos, family tree, real-time location sharing with an SOS button and notifications.
1.2  Background and Related Systems
As digital technology is already being used by the families for staying connected, the first step in the project was to assess what capabilities the existing technologies have and where they fall short. WhatsApp, Telegram, and Facebook Groups are currently used by families for communication and media exchange. These apps are free, intuitive, efficient means of communication, however, they lack functionality for scheduling family events and creating a proper archive of family history, as media just get lost in the endless stream [1]. Scheduling tools tackle the issue from another side. Doodle and Google Calendar do wonders at organizing meetings, but these apps are designed for businesses and for public use and are alien to the family experience [2]. When it comes to memory management, Google Photos and iCloud provide reliable cloud backup, however, they operate as a storage service and fail to create any connection between the photo and the family event it describes [3].
At the same time, literature emphasizes the fact that applications intended to serve users of various age should be explicitly designed to address their needs in usability. The System Usability Scale is the standard lightweight tool used to assess the level of usability of a particular application intended for users of different ages [4], while ISO 9241-11 provides the definition of usability based on three parameters of effectiveness, efficiency, and satisfaction, which makes it suitable to use when dealing with an application intended for users aged from childhood to old age [5]. Nielsen’s usability heuristics helped to make interface decisions described later in this thesis [6]. Finally, prior research on intelligent scheduling and reminder systems shows that the use of timely and automated reminders can significantly increase engagement and prevent missed appointments [7], which proves the importance of integrating reminder function into the core of a family coordination system. Lastly, secure communication has become the minimum requirement for any application that contains personal information. JSON Web Tokens are the widely accepted solution for API authentication without session tracking [8], and WebSocket protocol helps to provide bidirectional, low-latency messaging required for a family chat [9].
It became clear that there is a gap among all the existing technologies and studies mentioned above: no application provides family event scheduling, memory archiving, reminders, chatting and accessibility for the elder population within the closed network. This gap inspired Family Connect project.

1.4  Business Needs
• Enhance family communication and prevent forgetting birthdays, anniversaries, and meetings.
• Store family photos, videos, and tales in one centralized, organized folder that is not spread out in various public sites and personal gadgets.
• Plan events together with the help of availability voting and RSVPs rather than discussion threads.
• Help to enhance the feeling of security in the family using live location tracking and single-tap SOS alerts.
• Met all generations of the family, including the elderly and children, thanks to different interface options and languages.
1.5  Scope
This project provides a fully functional full-stack system with a mobile application, a back-end server, and cloud infrastructure connecting both. The following functionalities are covered in the scope of this delivered project.
Authentication and family management: registration, login, token refreshing, invite code based family creation, and role promotion of the family creator to an administrator of the family.
Chat: real-time messaging with media attachment capability, replies, reactions, mentions, pinning, starring, editing, deletion, search, and typing indication.
Events and polls: ability to create events and invite guests to take part in them automatically, response and availability tracking for date availability polls with a deadline and voting.
Memories and albums: upload of pictures and videos to a family gallery, likes, member tagging, albums management and sharing.
Family tree: graphical family representation with different member roles and relations that can be edited by any member.
Location and SOS: live location tracking on a family map, and SOS alerts combining push notifications and real-time sockets.
Notifications: in-app and push notifications for messages, events, memories and SOS alerts, and broadcasting of notifications for admins only.
Accessibility and language settings: normal, minor, and elderly user interface, high contrast and reduced motion, and English, Tamil and Sinhala languages.
Some of the functionalities mentioned initially have been rescope throughout the year and are therefore not part of the scope of the project: end-to-end encryption and two-factor authentication (transport security, hashed passwords, and token refreshing have been implemented instead), voice guided navigation, cultural celebration calendar, automatic smart date suggestions (availability polls have replaced this functionality), server persistence of safe zones and trips history (only locally available), release on public Google Play and iOS build. This will be justified in Chapters 8 and 9.
1.6  Assumptions
All participating members have a smartphone (Android phone in the pilot version) with internet access and GPS.
The members are ready to use an invitation-only and private platform instead of the social media platforms for family interactions.
The free versions of the cloud services selected (MongoDB Atlas, Cloudinary, Firebase, Railway) will be enough for the pilot implementation.
One family per user account will be enough for the pilot version; multi-family membership is left for future work.
1.7  Constraints
One calendar year was the delivery period with at least 400 hours of development effort needed for the project.
Single developer involvement in design, development, testing, deployment, and documentation.
Constrained budget for the cloud, aiming at below LKR 3,000 per month in the pilot stage, excluding any possibility of premium infrastructure tiers.
Data privacy issues related to an application storing family pictures, information about children, and live locations.
Milestone deadlines for the university, namely proposal defense, midterm evaluation, thesis draft, and final submission, being inflexible.
1.8  Objectives
Five goals were outlined by the proposal, which served as a point of reference for evaluation during the entire course of the project.
Establishing exclusive family groups that are safe from external influence and have role-based access control.
Employing family-centric scheduling and reminders with the use of availability polling and notification-driven prompts.
Creating an exclusive family archive where photos, videos, and stories can be archived and organized into albums.
Making sure that all age groups are accommodated through multilingual support (English, Sinhala, Tamil) and elderly-oriented interface option.
Conducting usability and security tests for the proposed platform.
1.9  Success Criteria
In terms of the charter, success was achieved by the following conditions. These criteria were assessed in chapter 8.
All essential functionalities – authentication, family management, chat, events with polling, memories with albums, family tree, location with SOS and notifications – are working in end-to-end manner with the production backend.
There is a verification of the isolation of the family data so that none of the authenticated user could read or change the records of another family.
The solution meets the results of structured test programme of all REST endpoints and real-time layer without any unresolved critical or high-priority defects.
The backend is hosted on a public cloud and the client is deployed and executed on the physical device of Android OS.
The interface is adaptive for elders, minors, and general users and the three language packs are in place.

Requirement gathering was done at the proposal stage through analysis of current platforms and needs of a multi-generational family, and the requirements prioritized with MoSCoW methodology into categories of must have, should have, could have, and won’t have. All the must haves - security of the family groups, scheduling using polling, memory archive, and the accessibility modes - remained intact throughout the development process. The could haves, such as voice navigation and the cultural calendar, were discarded because of real-time and safety modules being more useful and difficult than expected.

2.7  Security Requirements
As this system contains family photographs, kids’ stories, and actual addresses, security constraints were considered as first-class citizens, and explicitly tested for that matter. Never should passwords be saved or returned in plaintext; they should be hashed with bcrypt using the factor of 12 prior to saving. Each and every secure route requires a JWT bearer token [8]; moreover, invalid accounts should be denied explicitly with a status code. Refresh tokens need to be rotated on each and every invocation, which means that in case of any theft, a token will have only one use left in its lifetime. Any access to the family information has to be filtered on the server side using the requesting user’s family id; this is a multi-tenant isolation policy with zero tolerance for exceptions. Role-based access control is supposed to restrict access to all the destructive actions like editing events and albums, deleting messages, closing polls, broadcasting notifications, and modifying the tree. The API needs to apply some kind of rate limits (100 requests per 15 minutes in general and 20 per 15 minutes at login/registration). Uploads need to be limited by allowed file formats and max sizes before reaching the storage.
2.8  Quality Attributes
There were four qualities influencing daily decision-making during implementation. Consistency – every REST answer uses one envelope – an indication whether it is successful or not, along with data or human-readable text. Observability – backend uses winston logging system with HTTP lines in morgan format, and the health check shows the environment and timestamp. Testability – business logic is implemented in controllers using thin routes, which is the reason why we managed to cover 122 live REST tests by script. Inclusivity – accessibility is not an add-on on screens, but a context for each one of them.
3  Analysis and Design
This chapter describes the design of Family Connect: the overall architecture of the application, the behavioral models that characterize the interaction between users and components, the designs of the database and API, the architecture of the user interface from the client’s side, and the justification for the important design choices. These models reflect the actual design of the system; when an initial design has been changed during implementation, this is indicated.
3.1  Overall Architecture
The Family Connect system has a client-server design pattern with an extra real-time component. The mobile application is responsible for presentation and local state, the server-side is responsible for business logic, validation and interaction with external systems, and a Socket.io connection works concurrently with the REST API in the part of communication where request-response delay will be perceived by users, which includes chats, location tracking, and SOS messages.
Figure 3.1: High-level system architecture of Family Connect
Client application architecture consists of five layers inside. Screens use a design system; global states exist in React contexts (authentications, family, network and UI mode); feature modules access the data via their own hooks using corresponding domain services; transport uses Axios for REST and singleton Socket.io for events; and persistence divides secrets and preferences — where JSON Web Tokens live in SecureStore, and the rest is stored in AsyncStorage.
The backend has a traditional layering. Routes map URLs to controllers; controllers implement logic and call Mongoose models; JWT verification, validation of file uploads and errors processing occur inside middleware. There are three additional external services: MongoDB Atlas used for persistence, Cloudinary for storing and delivering media content and Firebase Cloud Messaging for push notifications. Thanks to this separation each service was testable independently — in particular, the QA framework used a local MongoDB instance with no Cloudinary credentials at all.

3.5  Database Design
The data layer employs MongoDB using Mongoose schemas. The document data model is appropriate for this domain, since the objects like events have naturally nested collections (guest list with status per each guest, voting options with vote arrays), which would require multiple join tables using the relational approach. Figure 3.6 presents ten collections and their relations.
Figure 3.6: Data model — collections, key fields, and relationships
Two general design principles permeate all the designs in the model. The first principle is that each family scoped document includes a family reference property, and each backend lookup is constrained by it. This is how multi-tenancy isolation is achieved; it is proven by the security test cases presented in Chapter 5. The second principle is duplication of information storage for the sake of two different goals: the Family document holds the members array for quick enumeration, whereas the FamilyMember collection holds per-member information – role, nickname, and relationship type that governs the family tree creation. Testing revealed the most critical bug of the project, because this duplication was broken during the creation process (Chapter 5, BUG-C1).

3.7  UI Architecture and Design System
The interface is based on a token-based design system instead of per-screen styling. Colors, typography, spacing, radii, shadows, and animation curves are tokenised once, and a set of primitives (buttons, cards, chips, text fields, dialogs, loaders, page headers) consumes those tokens. The composition happens at the screen level by combining primitives instead of restyling raw components. The benefit is that one can implement one function to resolve themes and reskin the whole app: switching to dark mode, high-contrast mode, or elder mode updates tokens, and everything gets restyled.
Accessibility is designed as a policy instead of just decoration. UIModeContext stores the current mode (standard, minor, or elder), resolved theme, and reduced motion flag provided by the OS; an accessibility policy module contains the set of rules regarding what minor mode is not allowed to do (upload media, show the invite code, share location, activate SOS); and screens check the policy by calling a hook instead of using conditional statements. In elder mode, the font scale becomes 1.24, touch target size 60 pt, and other UI elements get scaled as well. Internationalisation is implemented in the same way: an I18nProvider loads English messages on initialisation and Tamil or Sinhala when needed, provides a translation method with param interpolation, and saves the user preference.
3.8  Navigation Flow
Navigational system is based on React Navigation 7, using two-tier architecture, where at the application level there’s switching between either authentication stack or authenticated tab navigator and a stack navigator for each feature tab (see Figure 3.7). Deep linking from notifications is processed into the stacks – clicking on a message push navigates to the chat, SOS push navigates to the map. 3.9  Design Decisions
These decisions had the biggest impact on the final system. They are documented along with the alternative that was considered and the reason why the chosen option prevailed.
client-server architecture with a thin client rather than device-centric approach. Authorisation and validation occur entirely on the server side since the family app does not trust the device and a modified client still should not be able to access the other family’s information. The client side becomes responsible only for presentation, caching and interface optimistic updates.
Socket.io over raw WebSockets. Real-time component requires rooms (one per family), authentication, auto-reconnect functionality, and graceful degradation, and all of those are provided by default by Socket.io library on both the server side and the React Native client side [9]. Doing all that manually with raw WebSockets did not bring any additional benefits.
JSON Web Token with refreshing mechanism over the session-based system. Stateless access tokens make the API horizontally scalable and do the same job whether you are communicating via REST API or via socket handshake [8]. Rotations ensure that a compromised refresh token can be used only once. The drawback is non-immediate invalidation of an access token, documented in Chapter 8 as a hardening decision.
document-based model in MongoDB instead of relational database. Aggregates of the domain (events with guests and polls, messages with reactions and read list) map one-to-one with documents in MongoDB and schemaless iteration turned out to be useful during a year of evolving requirements [11]. The downside is that you have to be careful with some relational thinking, like dual member entities, and this price is paid just once in BUG-C1.
Cloudinary over uploading media files to the application server. Offloading storage, media processing and CDN distribution allows you to keep large binary files away from the API process and turn file upload validation to a declarative list of allowed types and maximum sizes. Free tier fit our budget constraints perfectly.
module pattern on the client. Since every feature consists of the same elements - navigator, data hook, service - the whole application codebase is uniform in its structure and features similar load, error handling and refresh mechanisms. Moreover, hooks implement one of the most important performance lessons: real-time events update the state in place, without re-fetching the module’s data (see Chapter 5 for details about refetch storms we experienced twice).
Design tokens and centralized accessibility settings. Accessibility modes were the key objective of this application, not something you implement at the end of the development, and only by implementing it as the context consumed by each screen we managed to create 3 modes and 3 themes in 3 languages within 73 screens.

4.2  Backend Implementation
4.2.1  Application Core
Server.js is the entry point that glues all the pieces together by creating the Express app and HTTP server, mounting Socket.io on the same port, connecting to MongoDB, initiating Firebase, adding security middleware (Helmet, CORS with an explicitly specified list of origins allowed, JSON body size limitation of 10 MB), registering the two rate limiters, mounting ten groups of routes at '/api', and ending with 404 handler and the main error handler. Health endpoint provides information on the environment used and current timestamp. Explicit handling of shutdown signals (SIGTERM and SIGINT) closes HTTP server prior to exit. Unhandled promise rejection is logged but not allowed to crash the service.
Special mention should be made of the error handler because it incorporates a safety measure which has been validated in tests – Mongoose specific error class is converted into the appropriate status code (422 for validation error, 409 for duplicate keys and 400 for invalid ObjectId), and in production mode response body contains no data except the sanitized message. The only systemic issue regarding this design found during tests relates to some controllers which catch exceptions inside their own try/catch, leading to returning 500 status code in case of malformed IDs rather than passing the CastError up to the main handler. The defect is marked BUG-L1 in Chapter 5.
4.2.2  Authentication and Family Management
Registration and logging in operations are located in authController.js. Passwords are hashed using Mongoose pre-save hooks via bcrypt library with salt cost 12, while the schema declares the password field as select:false in order to avoid returning it in a default query. During login, the system issues two tokens - short-term access token and a refresh token, the hash of which storage location is database. Each time refresh token is issued, server checks if the token that is being used matches the saved token, creates a new pair and replaces the stored token, making each refresh token usable only once. Logging out revokes the stored token. Protect middleware is responsible for verifying the bearer token of each protected route, loading the user and refusing deactivation access to 403.
Family management module (familyController.js) creates cryptographically secure invite codes, promotes the creator to admin status, and implements the family rules of membership starting from FR-05 to FR-08 - single-family-per-user constraint, double-joins prohibition, and leaving restriction for admins. After the discovery of the family-tree bug described in Chapter 5, functions create, join, and leave were expanded to take care of FamilyMember entity, together with embedded members array - upsert during creation and joining and delete during leaving.
4.2.3  Events, Polls, Memories, and Albums
The event controller handles auto-invite logic by taking a snapshot of the family membership in the form of the embedded guest list where each entry contains a pending status that the RSVP controller endpoint changes to accepted, declined, or maybe. Authorisation determines whether read or write access applies: any member reads events within a family, but only the author or an admin can modify or delete. The polls collection is related to the event and is handled separately by the controller which enforces that there are at least two choices, one choice per user for every choice set, time limit expiration, choice close by owner or admin, and writing of the chosen option to the event.
Memories are uploaded through multipart POST request. Using Multer, the file is streamed to Cloudinary using storage adapter per route; memory uploads support popular image and video types up to the configured size limit with avatars limited to 5 MB size; chat uploads allow additional audio and document types. The controller then writes the CDN url along with the caption, tags, and an empty likes array; toggling likes is idempotent. Albums hold memories, have a media counter and cover image, and can be shared via the generated link; add-media, update and delete album operations require the owner-or-admin authorisation logic.
4.2.4  Chat
Chat is the most extensive realm. The REST part of the application (chatController.js) offers paginated chat history capped at 200 messages, valid target replies within the same family, own only edition, own/admin deletion, emojis support, pins with timestamp and author, user star function, and text search. The search feature was an issue addressed in the security update: since the user input goes into MongoDB as a regular expression, metacharacters are safely escaped prior to search – the very update which resolved a ReDoS and regex injection issue identified during the release audit.
4.2.5  Location, SOS, and Notifications
One document per user is upserted by the location controller — the map requires just the last known location for each user — and the validation of coordinates is performed. Any updates that have been submitted over REST API are then sent to the family’s socket room via the shared Socket.io object, which is how all members’ maps stay up-to-date without polling. SOS endpoint utilizes the same principles: validate, save, notify over both channels. The notification service (services/notificationService.js) creates a new Notification document for each recipient and sends out FCM multicast using their registered device tokens; it is a fire-and-forget action within controllers to ensure that neither a delayed push nor push provider errors would hinder either chat or SOS notifications; and it cannot be used by anyone but admins since the test revealed that any family member could push to the whole family (BUG-H2).
4.3  Real-Time Layer
The socket server (socket/socketServer.js) validates each connection using the same JWT secret as the REST API, where the middleware checks the token, fetches the user, and refuses any connection without it, with an invalid or deactivated one. On connection the socket joins the room of its user's family, and all the broadcasts are scoped to this room - the system ensuring that all the real-time communications follow the same principle of isolation as the database queries. Handlers are implemented for message sending (persist, set sender info, acknowledge sending to the sender, broadcast message in the room, notify absent members), typing indicators, and read receipts. Handlers being defined by each connection, Socket.io will take care of cleaning them up on disconnection; the audit found no problems related to listener and room leaks.
On the client side, only one connection per session is opened after logging in, which is distributed via the small relay system: hooks subscribe to the events relevant to their functionality, and the relay unsubscribes from the server connection when there are no more listeners left.

4.5  Security Implementation
Security measures cut across both levels and were validated using test cases designed specifically for this purpose. On the server side: bcrypting passwords, JWT verification for all secured endpoints and on the socket connection, refresh token rotation, querying by family, authorization for destructive actions, two rate limiters, Helmet headers, a list of origins in CORS headers for production, allowing upload of only certain file types with a size limit, escaping search parameters, and sanitized error messages. On the client-side: Tokens stored exclusively in SecureStore and not in AsyncStorage nor in code, the API origin in production locked in configuration, and logging removed from release build configurations. These remaining hardening measures are recorded in Chapter 8.
5  Testing
There was just one rule that was applied throughout testing process and it was the following: nothing works until proven otherwise. Finally, the verification program checked all of the REST endpoints and the real-time layer in live manner against a running server and a real MongoDB database, supplemented the live tests by code review of the mobile client and then fixed issues only when they were reproduced. The chapter will describe the approach, harnesses used, results achieved, bugs fixed, evidence of regression and user acceptance test.
5.1  Testing Strategy
Verification took place in two stages. Release audit stage consisted of exhaustive examination of all backend files (47) and mobile source files (~311), tracking users' interaction with code, cross-checking paths with controllers and documentation, and verifying the Android build configuration. Then, a final stage with live testing included running 122 automated REST tests and 7 automated Socket.io tests against a server started up with a local MongoDB database, and 51 additional cases testing mobile behavior, build configuration, navigation, and regressions by inspecting code with file-and-line details. This ratio stems from the advantages of each type of verification — live execution proves behavior, while code verification tests device-bound behaviors (push delivery, GPS, gestures) impossible to test in headless mode.
Unit-level verification (validation rules, authorization rules, model constraints) was done via API calls rather than in isolation, which is a practical approach in a one-person project, where every unit is accessible via API endpoint, and test via HTTP calls tests routing, middleware sequence, and serialization. Integration testing was implicit in this design, since each test covers the route, middlewares, controller, model, and database. End-to-end user acceptance scenarios presented in Section 5.6 served as a basis for system testing, while manual exploratory testing on the Android emulator and physical device was done during the whole development process, guided by screen-specific QA checklist stored in the repository.
5.2  Test Environment and Harnesses
The live setup comprised Windows 10, Node v24.16.0, MongoDB as a local service having the family_connect database, and the backend at http://localhost:5000 with relaxed rate limits to account for testing loads. There are three scripts in backend/scripts that allow reproducing the application: qaLiveTest.js runs 122 REST tests including happy paths, failed validations, authentication and RBAC checks, tenant separation, injections, and token handling; qaSocketTest.js tests the socket handshake, broadcasting in a room, REST to socket routing, typing, and refusing non-authenticated connections; and qaGenerateXlsx.js produces the full list of tests. 6  Deployment
The following chapter provides the steps taken by the system for migration from developer's computer to user: the development process, the production setup for the backend in Railway, management of configurations via environment variables, the mobile build process, and the versioning practice for releases.
6.1  Development Environment
The development environment uses nodemon for the backend with local MongoDB and the client through Expo. There are two networking issues to be aware of regarding Android – the emulator can connect to the host’s API at http://10.0.2.2:5000, while the phone will need adb reverse port forwarding; both have been included in the committed scripts. The seed scripts generate a user for development along with demo data on request, and this decoupled the development of modules from manually creating data.
6.2  Backend Production Deployment
The backend is hosted on Railway via mobile-connect-app-production.up.railway.app [21]. The Node application is built from the repository by Railway and is provided secrets via its environment variables configuration, so no secret gets into the code or source control in any way. There are two key platform features that motivated the selection of Railway: WebSocket over the same HTTPS port, enabling Socket.io to operate from the same REST origin without any additional infrastructure; and default TLS termination, making all connections go over HTTPS and WSS. In production, CORS policy restricts access to only one configured client origin and the production server itself – the requests without Origin header (mobile apps) get through, other origins from browser are rejected.
6.3  Configuration and Environment Variables
All configuration is externalised. The repository ships a documented .env.example; Table 6.1 lists the variables and their roles. Development and production differ only in these values the codebase is identical.

6.4  Mobile Build Process
The project is built using Expo Application Services via three eas.json profiles: development (development client, internal distribution), preview (installable APK for testers), and production (Android App Bundle for the store). The app.json includes the native configuration applied at the build stage – the application id com.familyconnect.mobile, version code, permissions to use camera, microphone, and location with the description for the end-user, Google Maps API key, notification icon and color, and adaptive launcher icon. The build audit has proven that expo prebuild produces the fully coherent native project according to the configuration above.
There is one deliberate release blocker – the lack of production signing keystore leads to using debug keystore for release builds. The produced artifact is installed for demonstration and side-loading as per the project requirements, while the Google Play Store submission needs to produce the actual upload keystore or delegate signing to EAS (account-level operation in the list of future actions). Also, the build audit captured the precise commands to reproduce the builds both for local Gradle and EAS cloud environments.
6.5  Release Versioning and Rollback
Releases adhered to a release candidate policy, with rc.1 being the completion of the map module, rc.2 being the production hardening phase, and rc.3 being the build that was ready for final testing, and 1.0.0 would follow only upon QA approval. Rollbacks are very simple in both layers: Railway rolls back to an earlier image, and since Mongoose works under a schema-on-read architecture, there are no database migrations that need to be rolled back. Earlier versions of the mobile app can always be released provided that the API is still backward compatible, something ensured by the envelope architecture.

Every iteration ended with the module fully wired up, from schema to endpoints to screens, not the whole set of backends separately from the screens, and that proved very useful at the progress review, where we demonstrated authentication, families and initial flows of events.

Verified breadth. The ten components of the product from real-time chat to SOS integrate well and operate together as part of one product, while the verified truth behind the claim comes from running each endpoint in the live environment.

8.3  Weaknesses and Limitations
The assessment would not be complete without the other column. The obvious shortcoming is the lack of internationalization: the engine and all three bundles are present, however, the feature UIs are presented as hardcoded in English, which means that if an elder speaks Tamil or Sinhala, then the feature will provide a localized UI around the English content. Offline mode notices disconnection but fails to reproduce unsuccessful mutation because its domain processors of the queue have never been connected. Debug signing of the release artefact means that the app will not reach Google Play Store until there is a proper key store. Access tokens expire in 30 days and cannot be invalidated before the expiration, refresh tokens are stored unhashed, and the development CORS proxy has to be stripped from the release build — these are the security items that are blocking any untrusted deployment. Continuous integration process lacks automated unit-test suites, even though live tests are reproducible and run on demand. Last but not least, the assessment has boundaries: no subjective satisfaction study with families was performed during the year.

8.4  Performance Evaluation
Performance tuning focused on areas where the family application was detrimental to performance: unnecessary network round trips and unconstrained queries. The two types of storm that were discovered through audit analysis - full chat history refetches and full location refetches for each socket event - were fixed. Hot paths have indexes (messages based on family and time, starred lookup, locations based on family and recency, albums based on family). Chat history is restricted to 200 messages per request. Acknowledged and acceptable for the current scale: query for events and memory list, non-batched write and push notifications per recipient, independent fetching of family locations in dashboard, family and map modules. All flagged for future version but none is visible at family scale.
8.5  Security Evaluation
The security tests tried attacks a malicious family member or outsider would perform: requests without a token, old tokens, altered signatures, tokens that used incorrect secrets, NoSQL operator injections in login, regular expression metacharacters in searches, cross-family access to events and memories, destructive actions not authorized as admin (which was since patched), and unauthorized connections using sockets. All were refused properly. The audit conclusion remains the true assessment: safe-by-default in an academic and beta version, but needing the specified hardening — short token expiry, hashed refresh tokens, strict CORS, admin guard — before release for production use. Testing was done in accordance with OWASP guidance on vulnerability types to check [22].
8.6  Overall Assessment
The production-readiness review rated the system 78/100 points in total (backend: 74, mobile: 86), with a conditional approval to proceed with both the FYP demo and internal beta straight away, but depending on four conditions for going live with the public launch. In the context of the problem statement from chapter 1, the system works exactly as intended – the family is able to move their conversations, plans, memories, and safety net into a single private space – and the gap between the two is clear to see.
9  Conclusion
9.1  Summary of Achievements
While most general-purpose platforms provide communication, planning, memories, and security in separate channels, this project aimed to offer a single private space where all can be accomplished. The end result of the development of the software during one academic year was Family Connect – a React Native app with an Express, MongoDB, and Socket.io backend – featuring ten modules: authentication with token rotation, invite-based family management, real-time chat with responses, reactions, pins, stars, and search, events with RSVP and attendance poll, memory storage with photo albums and tagging, a family tree, live location sharing and dual-channel SOS alert, notifications with in-app and push functionality, accessibility with three user interface options and three supported languages, and offline detection with graceful degradation.
It is a matter-of-fact statement to claim that the system works as expected. All REST endpoints and the real-time layer of the system were tested using a 180-case program which ended up with a 94.4% success rate and no unresolved critical or high-level defects; all ten acceptance test scenarios were successful; and the fixes for four important defects – crashing gallery, silent SOS, empty family tree, and an open broadcast of notifications – were detected by the program and proven successful after re-execution.
9.2  Reflection on Objectives
Four out of five charter goals were met in spirit, and the fifth in terms of its technical side. Insofar as the actual design differs from the proposed one, such differences were intentional and, upon reflection, correct: polls for availability worked as a scheduling primitive better than smart suggestions; minor-mode policy-based gates were an enforcement strategy that offered more child safety than approval workflow; and spending the saved time on chat and map/SOS functionality made these the two strongest selling points of the product. The true shortcomings lie in inadequate language support on feature screens and lack of a usability test — both are clear-cut, known problems that come next.
9.3  Limitations
These constraints, as mentioned throughout both chapters 5 and 8, have to be reiterated at once: feature-screen internationalization is partial; offline mutations are not played back; the release build is debug signed rather than store validated; access-token expiration time, refresh token management, and the development CORS proxy should be secured before external exposure; many list queries and notifications dispatching are not optimized beyond the level of family; automated testing is an ad-hoc testing harness and not CI; no evidence of usability in real families was collected yet.
9.4  Future Work
Every item listed below is tangible in the sense that there already exists either architecture for it or a known way to get there.
Final internationalisation step, consisting of moving the feature screens into the already existing translation engine – the bundles have keys.
Registration of per-domain processors of the offline queue to make sure that any changes done while being offline will play back once the connection is back – the architecture is shipped already.
Creating of production keystore or usage of EAS-managed credentials, matching version identifiers and submitting the signed bundle into Google Play store; doing an iOS build.
Hardening authentication for public production – reducing lifetime of access tokens, hashing refresh tokens and getting rid of development CORS fallback.
Moving safe zones, emergency contacts and trip history into server persistence and adding geofence notifications to them.
Batching of notification writes and FCM dispatching, paginating remaining list endpoints, adding secondary indexes.
Setting up CI for running an automated test suite on existing harnesses, so that every change goes through 129 live tests.
Conducting the prepared SUS-based family pilot for various age groups and incorporating results of it into an accessibility-oriented release.
Completing the family tree with full person and relationship CRUD operations based on server persisted legacy profiles.
The problem that this project discovered in the beginning of its year-long development process – absence of a unifying, private and cross-generational platform for family use – was quite genuine, and one year of engineering effort proves that it is solvable with proper work on infrastructure of a free tier. Thus, Family Connect enters the end of the year as a verified, deployed beta product with all the remaining steps to production documented and outlined, exactly how it should be at this stage of development.

Appendix B: Test Case Distribution and Reproduction
The single workbook holds 180 tests: 122 live REST tests (automated), 7 live Socket.io tests (automated), and 51 tests with verified code for mobile behavior, build, navigation, and regression. Test results: 170 passed, 6 failed, 4 partially passed (94.4% success). The program can be recreated based on the files in the repository: first launch MongoDB; then run the server in the backend directory with an increased rate limit in development mode; next run qaLiveTest, qaSocketTest, and qaGenerateXlsx files. Original results are kept in JSON format in docs/qa folder of the repository.
